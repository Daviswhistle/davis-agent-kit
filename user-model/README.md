# User Model

사용자와 함께 작업하면서 파악한 작업 방식, 품질 기준, 선호, 반복 피드백을 기록합니다.

목적은 에이전트가 사용자의 판단 기준과 작업 습관을 따라 사고하고 행동하는 데 필요한 정보를 구조화하는 것입니다. 사용자에 대한 사적 분석은 배제하고, 다음 작업의 품질에 영향을 주는 정보만 기록합니다.

## 기록 기준

기록해도 되는 것:

- 반복해서 드러난 작업 선호
- 결과물 품질에 직접 영향을 주는 기준
- 싫어하는 실패 유형
- 자주 쓰는 출력 형식
- 검증 방식에 대한 기대
- 공개 가능한 작업 흐름
- 투자 판단 방식
- 개인 선호와 작업 철학

기록하지 않는 것:

- 계정, 토큰, 키, 비밀번호
- 사적인 신상 정보 중 작업 재현성과 무관한 내용
- 근거 없는 성격 분석
- 사용자가 원하지 않는 사적 기록
- 비공개 원문 전문
- 제3자 민감정보
- 계약상 비공개 자료

## 관찰 계약

각 관찰은 `observations.md`에서 다음 필드를 사용합니다.

```text
ID: OBS-YYYY-MM-DD-NNN
상태: candidate | confirmed | promoted | superseded | retired
마지막 확인: YYYY-MM-DD
재검토: YYYY-MM-DD 또는 구체적인 재검토 조건

관찰:
근거:
작업에 미치는 영향:
승격:
- `현재 적용 중인 경로`
대체:
없음 | supersedes OBS-... | superseded_by OBS-...
확신도:
```

필드의 의미:

- `ID`: 날짜와 순번으로 만든 안정적인 관찰 식별자
- `상태`: 관찰이 운영 규칙으로 발전한 현재 단계
- `마지막 확인`: 사용자의 명시적 피드백이나 실제 작업으로 마지막 확인한 날짜
- `재검토`: 관찰이 오래된 전제가 되지 않도록 다시 확인할 시점이나 조건. 안정적인 `promoted` 관찰은 기본 6개월을 사용하고, 환경 의존성이 크면 더 짧은 기한이나 사건 기반 조건을 사용
- `승격`: 현재 관찰을 실행하는 실제 파일 경로. 과거에 존재했던 경로는 근거에 남길 수 있지만 이 필드에는 쓰지 않음
- `대체`: 관찰 간 대체 관계. 방향을 `supersedes` 또는 `superseded_by`로 명시
- `확신도`: 근거의 반복성과 명시성을 바탕으로 한 `낮음`, `중간`, `높음`

## 상태 전이

- `candidate`: 한 번의 사례나 아직 일반화되지 않은 피드백
- `confirmed`: 반복 확인됐거나 사용자가 일반 기준으로 명시했지만 실행물에 아직 반영되지 않은 관찰
- `promoted`: `AGENTS.md`, 가이드라인, 체크리스트, 스킬, 템플릿, 예시 중 현재 경로에 반영된 관찰
- `superseded`: 더 구체적이거나 현재 상황에 맞는 다른 관찰로 대체된 기록
- `retired`: 더 이상 적용되지 않으며 후속 기준으로도 유지할 필요가 없는 기록

상태는 한 방향으로만 증가한다고 가정하지 않습니다. 반례나 사용자 정정이 생기면 `promoted`도 `superseded`나 `retired`로 바꿀 수 있습니다.

## 승격 규칙

1. 새 관찰은 `candidate`로 기록합니다.
2. 반복 확인되거나 사용자가 일반 기준으로 명시하면 `confirmed`로 올립니다.
3. 작업 재현성에 도움이 되는 위치에 실제 반영한 뒤 `promoted`로 바꾸고 `승격`에 현재 경로를 기록합니다.
4. 새 관찰이 기존 관찰을 대체하면 양쪽 `대체` 필드에 방향을 남깁니다.
5. `재검토` 시점에는 근거, 적용 범위, 승격 경로의 현재성, 충돌 관찰을 확인합니다.
6. `promoted` 관찰의 승격 경로가 이동하거나 삭제되면 같은 변경에서 필드를 갱신합니다.

승격 위치의 역할:

- 전역 판단 기준: `AGENTS.md`
- 특정 작업 표면의 적용 지침: `guidelines/`
- 완료 전 검수: `checklists/`
- 반복 작업 절차: `skills/`
- 출력 형식: `templates/`
- 좋은/나쁜 사례: `examples/`
