# 게시 가능한 한국어 HTML 아티클

## 적용 범위

분석 원고를 단일 파일 HTML/CSS, CMS 붙여넣기용 코드, 미리보기 이미지로 완성할 때 적용한다. `writing-quality`는 주장, 한국어, 정보 구조와 시각적 의미를 소유한다. HTML/CSS 구현과 렌더링 검증이 실질적인 작업이면 `software-engineering`을 함께 적용한다.

## 완료 계약

- 초안이 아니라 그대로 게시할 수 있는 산출물을 만든다.
- 원고, HTML, CMS용 코드, 미리보기는 같은 최종 내용을 사용한다.
- 실행하지 못한 검증과 남은 위험을 구분해 기록한다.

## 1. 주장과 제목

1. 본문을 쓰기 전에 글이 입증할 핵심 주장을 한 문장으로 고정한다.
2. 제목은 가장 큰 명사나 가장 자극적인 대립보다, 본문이 보여 주는 변화·메커니즘·판단을 정확한 강도로 압축한다.
3. 병목이나 중심 변수가 이동하는 글에서는 정적인 `진짜 X는 Y다`보다 이동 방향과 시간을 보여 주는 제목을 우선 검토한다.
4. 본문이 인정하는 기존 변수까지 제목에서 부정해 거짓 양자택일을 만들지 않는다.
5. 제목을 지키기 위해 본문에 방어 문단이 필요해진다면 제목의 강도를 다시 낮춘다.

## 2. 한국어 독해 비용

- 각 문장은 새로운 사실, 인과, 판단, 전환 중 하나를 앞으로 보낸다.
- 글에 없는 반론을 먼저 세우고 본론으로 돌아가는 문장을 제거한다. 부정과 대조는 실제 오독을 막고 결론을 바꿀 때만 사용한다.
- 영어 자료를 재서술할 때는 사실과 논리 관계를 보존하고, 주어·문장 순서·용어는 한국어 독자의 이해 순서로 다시 구성한다.
- 원문의 부사와 추상 명사구를 직역하지 않는다. `전력회사별 전망을 종합해 추산했다`처럼 실제 행위와 결과를 직접 쓴다.
- 전문 용어는 해당 분야의 정확한 의미를 전달하고 독자의 판단을 줄일 때만 사용한다. 평이한 한국어가 더 정확하면 전문 용어를 제거한다.

## 3. 대중 대상 수치 표현

- 한국어 대중 원고와 수치 카드에서는 비교 기호보다 `이상`, `이하`, `초과`, `미만`을 사용한다. 예: `<3%`는 `3% 미만`, `>5년`은 `5년 초과`, `1,000%+`는 `1,000% 이상`으로 쓴다. 수식, 코드, 축 레이블처럼 기호가 실제 기능을 가질 때는 예외다.
- 숫자와 비교어는 하나의 의미 단위다. `이상`을 별도 배지, 버튼, 상태표시처럼 보이게 분리하지 않는다.
- 기간, 분모, 단위와 기준 시점을 숫자 가까이에 둔다.
- 영문 약어와 축약 단위는 한국어 독자의 이해를 빠르게 할 때만 유지한다.

## 4. 원자료의 의미 범위

숫자 자체와 함께 다음을 확인한다.

- 무엇을 집계했는가: 프로젝트, 용량, 기업, 응답자, 부하 신청 등 대상 집합
- 언제와 어떤 단위인가: 기준일, 기간, 누적·증분, 명목·실질, MW·MWh 등
- 어떤 성격인가: 관측값, 추정치, 전망, 시나리오, 모델 결과
- 어떤 상태인가: 제안, 의견 수렴, 표결, 최종 결정, 시행, 실제 효과
- 어떤 출처인가: 공식 통계·규제 문서, 논문, 기업 발표, 스트레이트 보도, 전문가 논평

요약 문구와 시각 카드도 원자료보다 강한 인과나 확정성을 만들지 않는다. 범위를 밝힐 때는 존재하지 않는 오해를 길게 부정하기보다 집계 대상과 상태를 직접 쓴다.

## 5. 시각적 의미

- 수치 카드, 흐름도, 비교 패널, 체크리스트는 각각 규모, 과정, 대조, 적용처럼 분명한 논리 기능을 맡는다.
- 장식만 늘리고 본문과 같은 말을 반복하는 블록을 만들지 않는다.
- 문법적으로 붙어 있는 요소는 시각적으로도 붙인다. 숫자와 단위, 수치와 비교어, 제목과 범위 문구를 임의로 분리하지 않는다.
- 배지, 버튼, 탭처럼 보이는 모양은 실제 상호작용이나 상태 의미가 있을 때만 사용한다.
- 한 줄 배열을 지키려고 글자를 과도하게 줄이거나 의미 단위를 찢지 않는다.

## 6. 실제 게시 환경에서 렌더링하라

1. 브라우저 전체 폭이 아니라 실제 아티클 컨테이너 폭을 기준으로 검증한다. 부모 폭이 뷰포트와 다를 수 있으면 컨테이너 쿼리나 그에 준하는 구조를 사용한다.
2. 목표 데스크톱 폭, CMS의 중간 폭, 좁은 모바일 폭을 모두 렌더링하고 실제 검사 폭을 기록한다.
3. 긴 한국어 복합어, 공백이 적은 항목, 숫자와 단위, 출처 레이블을 대표 스트레스 케이스로 둔다.
4. 가로 스크롤, 잘림, 겹침, 지나친 축소, 의도하지 않은 줄바꿈, 클릭 요소처럼 보이는 장식을 확인한다.
5. DOM 경계나 오버플로 검사만으로 끝내지 않는다. 최종 스크린샷을 사람이 읽어 문장과 시각 문법이 자연스러운지 확인한다.

## 7. 피드백 이후 전수 회귀 검수

- 사용자가 한 사례를 지적하면 그 표현만 바꾸지 말고 같은 실패 유형을 원고 전체, HTML, CMS용 코드, 미리보기에서 검색한다.
- 텍스트를 고친 뒤 HTML과 CMS용 코드의 내용이 같은지 비교하고, 최종 HTML에서 미리보기를 다시 만든다.
- 전체 검색과 재렌더링을 끝내기 전에는 `전체를 수정했다`, `문제가 남아 있지 않다`고 보고하지 않는다.

## 최종 증거

완료 보고 전에 다음을 남긴다.

- 최종 산출물 목록과 같은 버전임을 확인한 방법
- 원자료에서 다시 확인한 수치와 의미 범위
- 검색한 반복 오류 패턴
- 렌더링한 실제 컨테이너 폭
- 사람이 확인한 전체 또는 문제 구간 스크린샷
- 실행하지 못한 검사와 남은 위험
