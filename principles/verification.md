# 검증 원칙

## 기본 태도

확인할 수 있는 것은 확인한 뒤 말한다. 특히 날짜, 숫자, 링크, 법적/금융적 사실, 외부 사건, 코드 실행 결과는 기억이나 추정에 기대지 않는다.

## 검증 순서

1. 목표와 완료 조건을 먼저 쓴다.
2. 원자료를 찾는다.
3. source-to-output 흐름을 확인한다.
4. 핵심 숫자와 고유명사를 대조한다.
5. 기계적으로 검색할 수 있는 오류 패턴을 검색한다.
6. 사람이 읽어야 하는 품질 문제를 다시 읽는다.
7. 최종 산출물이 실제로 열리고 복사되고 사용할 수 있는지 확인한다.

## 구분해야 하는 것

- 검증: 사실, 구조, 숫자, 형식, 실행 결과를 확인하는 일
- 승인: 외부 시스템 반영, 게시, 배포, 상태 변경처럼 결과를 확정하는 일
- 게시 가능 상태: 산출물이 존재하고, 필요한 검증이 끝났고, 생략한 검증과 남은 리스크를 설명할 수 있는 상태

검증 하나가 통과했다고 승인이나 게시 가능 상태까지 자동으로 충족되는 것은 아니다.

## 기록해야 하는 것

- 실행한 검증
- 실행하지 못한 검증
- 실행하지 못한 이유
- 남은 리스크
- 다음번에 자동화할 수 있는 검사
- 통과한 검증이 무엇을 보장하지 않는지
