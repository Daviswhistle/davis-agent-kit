# Ponytail 최소화 원칙 반영 제안

이 문서는 Ponytail의 핵심 원칙을 기존 전역 규칙에 흡수하기 위한 제안이다.

AGENTS.md와 GEMINI.md 모두 `작업 시작 전 절차`와 `구현 원칙` 사이에 아래 섹션을 추가한다.

```markdown
## 구현 전 최소화 판단

코드를 작성하기 전에 아래 순서대로 먼저 판단한다. 목적은 성급한 축약이 아니라 불필요한 구현을 줄이는 것이다.

1. 이 기능을 정말 새로 만들어야 하는가? 기존 동작, 설정, 문서화, 운영 절차로 충분히 해결되는지 먼저 확인한다.
2. 표준 라이브러리로 해결할 수 있는가? 가능하면 표준 라이브러리를 우선 사용한다.
3. 운영체제, 런타임, 프레임워크, 배포 플랫폼의 기본 기능으로 해결할 수 있는가? 가능하면 플랫폼 기본 기능을 우선 사용한다.
4. 이미 설치된 의존성이나 기존 프로젝트 코드가 같은 문제를 해결하고 있는가? 가능하면 기존 것을 재사용한다.
5. 새 코드를 써야 한다면 가장 작고 명확한 형태로 구현한다. 단, 가독성, 검증 가능성, 오류 처리, 유지보수성을 해치면서까지 줄이지 않는다.

### 최소화 원칙

1. 명시적으로 요청되지 않은 추상화는 만들지 않는다.
2. 피할 수 있으면 새 의존성을 추가하지 않는다.
3. 사용자가 요청하지 않은 보일러플레이트, 범용화, 확장 포인트를 만들지 않는다.
4. 기존 코드 삭제나 단순화로 해결되는 경우 새 코드를 추가하지 않는다.
5. 단순하고 평범한 구현을 우선한다. 영리하지만 설명이 필요한 구현은 피한다.
6. 복잡한 요청에 대해서는 더 작은 대안으로 충분한지 검토한다.
7. 같은 길이의 표준 라이브러리 접근이 여러 개라면 경계값과 예외 처리가 더 정확한 쪽을 선택한다.
8. 의도적인 단순화에는 `ponytail:` 주석을 남긴다. 해당 단순화의 한계가 있다면 한계와 향후 확장 방향을 함께 적는다.

예시:

```python
# ponytail: 현재 입력 규모에서는 O(n²) 비교가 더 단순하고 충분하다.
# 데이터가 수만 건 이상으로 늘어나면 key 기반 인덱스로 바꾼다.
```

### 최소화하면 안 되는 것

아래 항목은 코드량을 줄인다는 이유로 생략하거나 약화하지 않는다.

1. 신뢰 경계의 입력 검증
2. 데이터 손실을 막는 오류 처리
3. 보안 관련 검증과 권한 확인
4. 접근성
5. 실제 하드웨어, 시간, 센서, 외부 API처럼 이상적 스펙과 실제 동작이 어긋날 수 있는 부분의 보정
6. 사용자가 명시적으로 요청한 동작
7. 미래정보 사용 방지, 데이터 신선도 확인, 결측 감지, 재현성 보장처럼 백테스트와 자동화 작업의 신뢰성을 지키는 장치
8. 실패 시 조용히 잘못된 결과를 내지 않도록 하는 검증과 로그

비사소한 로직을 추가하거나 수정했다면 가장 작은 실행 가능한 검증을 하나 남긴다. 프레임워크, 대형 fixture, 과도한 mock을 새로 만들 필요는 없지만, 해당 로직이 깨졌을 때 실패하는 테스트, assert 기반 self-check, 또는 국소 검증 명령 중 하나는 있어야 한다. 단순한 한 줄 변경이나 명백한 문서 수정에는 별도 테스트가 필요하지 않을 수 있다.
```

## 함께 확인할 사항

GEMINI.md에는 기존 문장 깨짐으로 보이는 부분이 일부 있다. 실제 AGY CLI용 규칙을 수정할 때는 최소화 원칙 추가와 별도 커밋으로 문장 깨짐도 정리하는 것이 좋다.
