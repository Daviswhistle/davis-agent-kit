# 프롬프트 변경과 GPT-5.6 이행

## 목적

작동 중인 프롬프트 스택을 더 짧고 명확하게 만들되, 어떤 변경이 실제 품질 변화의 원인인지 확인할 수 있게 한다.

## 남겨야 할 계약

복잡한 프롬프트에는 필요한 항목만 짧게 둔다.

1. 결과: 사용자가 받게 될 산출물이나 완료 상태
2. 성공 기준: 최종 응답 전에 반드시 충족해야 할 조건
3. 제약: 안전, 권한, 증거, 사업 규칙, 부작용 경계
4. 도구: 맥락에 따라 달라지는 라우팅과 필수 선행 조회
5. 출력: 필요한 형식, 언어, 길이, 필드
6. 중단 조건: 언제 답하고, 재시도하고, 질문하고, 좁혀서 보고할지

결과를 정의한 뒤 모델이 효율적인 경로를 선택하게 한다. 안전이나 필수 필드처럼 진짜 불변 조건이 아닌 판단 문제에는 무조건 규칙보다 선택 기준을 쓴다.

## 줄이는 순서

다음 항목을 한 묶음씩 제거하거나 통합하고, 매번 같은 평가를 다시 실행한다.

- 같은 의미를 반복하는 문장
- 행동을 바꾸지 않는 스타일·절차 설명
- 결과를 바꾸지 않는 예시
- 모델이 안정적으로 수행하는 과정의 과도한 지정
- 현재 작업과 관계없는 도구와 설명

사용자에게 보이는 결과, 성공 기준, 중단 조건, 안전·권한·증거 제약, 맥락 의존 라우팅, 출력·검증 계약은 유지한다.

## 변경 절차

1. 현재 모델과 reasoning effort로 대표 과제의 기준선을 기록한다.
2. 프롬프트, 도구 집합, 모델, reasoning effort를 동시에 바꾸지 않는다.
3. 한 번에 하나의 의미 있는 변경 묶음만 적용한다.
4. 동일한 입력과 평가 기준으로 다시 실행한다.
5. 품질을 통과한 경우에만 토큰, 비용, 지연, 도구 호출 감소를 개선으로 인정한다.
6. 회귀가 생기면 실제 실패 trace에서 원인을 찾고 가장 작은 지침만 보강한다.

## 대표 평가 기록

최소한 다음 필드를 남긴다.

```text
case_id
model
reasoning_effort
prompt_version
skill_invoked
rubric_score
hard_failure
input_tokens
output_tokens
tool_calls
elapsed_time
grader_notes
```

정적 계약 테스트는 파일·라우팅·필수 문구의 구조적 회귀를 막는다. 실제 출력 품질은 새 대화나 격리된 실행에서 대표 과제를 수행하고 별도로 평가한다. 둘을 같은 검증으로 간주하지 않는다.

## 검색과 도구 중단 조건

1. 핵심 요청을 필요한 근거로 답할 수 있으면 중단한다.
2. 필수 사실, 소유자, 날짜, ID, 원문, 인용이 없을 때만 추가 조회한다.
3. 표현을 다듬거나 선택적 예시를 늘리기 위해 다시 검색하지 않는다.
4. 결과가 비어 있거나 지나치게 좁으면 의미 있는 폴백을 한두 번 시도한다.
5. 필요한 근거를 끝내 얻지 못하면 부재를 사실상의 부정으로 바꾸지 말고 결론 범위를 좁힌다.

## reasoning effort

1. 기존 설정을 기준선으로 보존한다.
2. 같은 설정과 한 단계 낮은 설정을 대표 과제에서 비교한다.
3. `high`나 `xhigh`는 평가에서 의미 있는 향상이 확인될 때 사용한다.
4. `max`는 가장 어려운 품질 우선 작업에만 사용하며 전역 기본값으로 두지 않는다.
5. reasoning effort를 높이기 전에 성공 기준, 의존성, 라우팅, 검증 루프의 누락을 먼저 확인한다.
