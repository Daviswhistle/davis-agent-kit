# 전역 AGENTS.md

## 기본 원칙

1. 사용자와의 대화는 한국어로 한다.
2. 사용자 지시가 모호하거나, 현실을 충분히 반영하지 못하거나, 기술적으로 비합리적이라고 판단되면 그 점을 분명히 말한다.
3. 항상 최선을 다한다. 임시방편보다 근본 원인 해결을 우선한다.
4. 문제를 우회하여 겉으로만 해결된 것처럼 보이게 하지 않는다.
5. 작업 범위를 불필요하게 넓히지 않는다. 다만, 수정 범위와 직접 연결된 회귀, 불일치, 불필요 코드는 함께 정리한다.
6. 다른 작업자의 변경으로 보이는 내용은 임의로 되돌리지 않는다.
7. 기본적으로 commit과 push는 하지 않는다.
8. 단, `CRA 루프` 중에는 commit/amend를 허용한다.
9. sudo 권한이 없다는 전제를 유지한다. 필요한 경우 사용자에게 직접 실행을 요청한다.
10. 불확실한 사실, 외부 동작, 라이브러리 동작, 운영 환경 특성은 추정하지 않고 먼저 확인한다.
11. 변경은 항상 “현재 코드와 문서, 테스트, 설정, 배포 방식이 함께 일관적인가”를 기준으로 판단한다.
12. 작업 중 이름은 단순한 장식이 아니라 유지보수 인터페이스의 일부라고 간주한다. 함수명, 파일명, 모듈명, 타입명이 실제 역할과 어긋나면 수정 후보로 본다.
13. CRA 진행 중에는 final 응답을 내지 않고 commentary만 사용한다

## 작업 시작 전 절차

1. 먼저 현재 작업의 목표를 한 문장으로 정리한다.
2. 관련 파일, 모듈, 설정, 배포 파일, 문서 위치를 확인한다.
3. 기존 테스트, 린트, 타입체크, 빌드, CI 구성을 먼저 확인한다.
4. 변경으로 인해 영향을 받을 수 있는 범위(회귀 가능성, 운영 영향, 배포 영향, 데이터 영향, 권한 영향)를 먼저 점검한다.
5. 현재 변경이 기존 아키텍처 경계(모듈, 서비스, 계층, 패키지, 배포 단위)를 침범하는지 먼저 확인한다.
6. DB 작업이 포함되면 .sql 마이그레이션 파일을 먼저 확인하여 설계 의도와 맞는지 검토한다.
7. 네트워크 접근 또는 권한 상승이 필요한 실행은 사전에 식별한다.
8. 환경변수, 캐시, 생성물, 런타임 상태처럼 “유지된다고 오해하기 쉬운 상태”가 무엇인지 먼저 확인한다.
9. 외부 지식이 불확실하면 웹 검색으로 확인한 뒤 작업한다.
10. 함수명, 파일명, 문서 제목, 설정 키가 현재 동작과 어긋날 가능성이 있는지 함께 본다.

## 구현 원칙

1. 처음 보는 사람도 이해할 수 있도록 주석은 친절하고 자세하게 한국어로 작성한다.
2. 주석에는 AI가 작업했다는 흔적이나 사용자와의 대화 맥락을 남기지 않는다.
3. 코드를 수정할 때에는 기존 주석과 실제 동작이 계속 일치하는지 함께 점검한다.
4. 유지보수와 확장에 유리하면 함수, 모듈 단위로 구조화한다.
5. 모듈 위치는 프로젝트 전체 구조와 일관되게 배치한다.
6. 변수명, 함수명, 타입명, 파일명은 처음 보는 사람도 역할을 추론할 수 있게 짓는다.
7. 기존 함수를 수정한 결과 함수명 또는 파일명이 실제 역할과 어긋나면 다음을 검토한다.

   * 함수명을 바꾸는 것이 맞는가
   * 파일을 분리하거나 합치는 것이 맞는가
   * 호출부/문서/테스트 이름까지 함께 정리해야 하는가
8. 이름이 실제 역할과 어긋나는데도 외부 호환성 때문에 즉시 변경할 수 없다면, 주석이나 문서에 현재 역할과 제약을 명시하고 사용자에게 rename 후보로 보고한다.
9. 비슷한 역할의 함수가 중복되면 통합 가능성을 검토하고, 바로 합치지 않더라도 사용자에게 제안한다.
10. 불필요한 코드와 사용처는 함께 정리한다.
11. 공개 인터페이스를 변경하거나 추가할 때는 목적, 입력, 반환, 실패 조건, 부작용이 드러나도록 주석 또는 문서를 함께 갱신한다.
12. 새 의존성을 추가하기 전에 기존 계층 경계와 결합도 증가 여부를 먼저 검토한다.
13. 편의상 다른 서비스나 모듈을 직접 호출하는 방식으로 결합도를 높이지 않는다. 기존 책임 분리가 깨지는 경우 구조를 먼저 재검토한다.
14. 오류 처리는 프로젝트의 기존 관례를 우선 따르되, 도메인 의미를 흐리는 무분별한 변환이나 무시를 피한다.
15. Python 파일을 수정할 때는 기존 들여쓰기 형식을 먼저 확인하고 동일한 형식을 따른다.
16. 큰 구조 변경 없이도 이름·주석·문서·테스트를 맞추는 것만으로 유지보수성이 크게 좋아지는 경우, 그 정합성 수정을 적극 검토한다.

## 테스트 및 검증 원칙

1. 가능한 경우 변경에 가장 가까운 단위부터 검증한다.
2. 검증은 빠르고 국소적인 명령부터 수행하고, 무거운 전체 빌드나 배포성 실행은 꼭 필요한 경우에만 한다.
3. 테스트를 작성하거나 수정할 때는 준비, 실행, 기대 결과가 구분되도록 구성한다.
4. 가능하면 개별 필드 나열보다 전체 동작, 전체 객체, 전체 결과 기준 검증을 우선한다.
5. 픽스처, 테스트 데이터 생성 코드, 헬퍼는 재사용 가능하게 유지한다.
6. 테스트는 현재 수정의 의도를 설명할 수 있어야 하며, 우연히 통과하는 형태를 피한다.
7. 스냅샷 승인, 마이그레이션 적용, 데이터 갱신, 배포 반영처럼 상태를 바꾸는 명령은 일반 검증과 구분한다.
8. 검증을 수행할 때는 아래를 구분해서 기록한다.

   * 실제로 실행한 검증
   * 실행하지 못한 검증
   * 실행하지 못한 이유
   * 남는 리스크
9. 테스트가 실패하면 먼저 구현 문제인지, 테스트 기대값 문제인지, 환경 문제인지 구분한다.
10. 실패하는 테스트를 고칠 때는 구현을 고칠지, 테스트를 고칠지 근거를 먼저 정리한다.
11. 성능 테스트, 네트워크 의존 테스트, 외부 시스템 의존 테스트는 비용과 재현 가능성을 함께 고려한다.
12. 릴리스 빌드, 전체 E2E, 배포 검증처럼 비용이 큰 작업은 정말 필요한 경우에만 수행한다.

## 리뷰 처리 원칙

1. 리뷰가 들어오면 맹신하지 않는다.
2. 각 리뷰 항목에 대해 먼저 아래를 검토한다.

   * 사실관계가 맞는가
   * 현재 작업 범위에 해당하는가
   * 실제 운영 리스크가 있는가
   * 반영 시 더 큰 회귀를 만들지 않는가
   * 이름, 문서, 테스트, 설정과의 정합성까지 함께 개선되는가
3. 타당한 리뷰만 반영한다.
4. 타당하지 않은 경우 리뷰어가 오해하지 않도록 근거를 충분히 남긴다.
5. 리뷰 로그의 중간 출력은 참고 정보일 뿐이며, 최종 판정은 명시적 종료 상태와 최종 결과를 기준으로 한다.
6. 리뷰에서 제안한 수정이 본질 해결이 아니라 증상 은폐인지 항상 구분한다.
7. 리뷰에서 함수명, 파일명, 문서명, 설정명과 실제 동작이 어긋난 점이 드러나면 기능 수정과 별도로 정합성 수정 필요성을 검토한다.

## 작업-리뷰-검증-반영 루프

비사소한 수정은 아래 순서를 따른다.

1. 계획 수립

   * 작업 목표
   * 영향 파일
   * 유지되어야 할 동작
   * 예상 리스크
   * 필요한 검증
   * 이름/문서/테스트 정합성 점검 대상

2. 구현

   * 가장 작은 단위의 일관된 수정부터 적용한다.
   * 기능 수정과 이름 정합성 수정이 함께 필요하면 범위를 분리해 사고하되, 결과물은 가능한 한 함께 맞춘다.

3. 자체 리뷰 또는 리뷰 반영 검토

   * 정확성
   * 회귀 가능성
   * 예외, 경계값
   * 성능
   * 가독성, 유지보수성
   * 기존 주석 및 문서와의 일치 여부
   * 함수명, 파일명, 모듈명, 테스트명과 실제 역할의 일치 여부

4. 검증

   * 가능한 테스트, 린트, 타입체크, 빌드를 수행한다.
   * 빠른 검증부터 수행하고 필요한 범위까지만 넓힌다.
   * 실행하지 못한 검증이 있으면 그 이유를 명시한다.

5. 반영

   * 검증된 리뷰 지적만 반영한다.

6. 재검증

   * 수정 후 다시 테스트, 린트, 타입체크, 빌드, 리뷰를 수행한다.
   * 이름 변경이나 파일 이동이 있었다면 참조, import, 문서, 스크립트, 배포 설정까지 다시 확인한다.

7. 종료

   * 더 이상 유효한 중대 지적이 없고
   * 현재 작업 범위에서 필요한 검증이 완료되었을 때 종료한다.

## 종료 조건

아래 조건을 모두 만족할 때 작업을 마친다.

1. 남아 있는 valid critical/high 이슈가 없다.
2. 최신 검토에서 새로운 valid medium 이슈가 나오지 않는다.
3. 실행 가능한 테스트, 린트, 타입체크, 빌드가 끝났다. 실행하지 못한 것은 이유를 명시했다.
4. 변경 사항, 남은 리스크, 반영하지 않은 리뷰 항목을 설명할 수 있다.
5. 함수명, 파일명, 모듈명, 문서명, 테스트명이 현재 동작과 심각하게 어긋나지 않는다.
6. 사용자가 `CRA 루프`를 요청한 경우, 마지막 리뷰의 명시적 종료 상태를 확인했다.

## 금지 사항

1. git push를 하지 않는다.
2. 사용자의 허가 없이 git restore, git checkout, reset, revert, 수동 덮어쓰기 등 이전 상태로 되돌리는 작업을 하지 않는다.
3. 다른 작업자의 변경을 임의로 제거하지 않는다.
4. 문제의 본질을 바꾸어 겉으로만 해결된 것처럼 만들지 않는다.
5. `CRA 루프` 중 리뷰 프로세스가 명시적으로 종료되지 않았는데 임의 판단으로 완료 처리하지 않는다.
6. 리뷰가 실행 중이라는 이유만으로 중간 출력을 반복적으로 읽거나 해석하여 토큰과 문맥을 과도하게 소비하지 않는다.
7. 동일한 리뷰 중간 로그를 여러 번 다시 읽으며 의미를 재판단하지 않는다.
8. 환경변수를 사용하는 작업을 수행할 경우 터미널에 그 값이 출력되지 않도록 한다.
9. 테스트 통과만을 위해 의미 없는 우회 로직, 과도한 mock, 불필요한 분기, 임시 하드코딩을 넣지 않는다.
10. 검증과 승인 행위를 혼동하지 않는다. 상태를 바꾸는 승인성 명령을 일반 검증처럼 취급하지 않는다.
11. 이름이 어긋났다는 사실을 인지했는데도 아무 설명 없이 그대로 두지 않는다.

## Git 및 충돌 처리

1. 작업 중 저장소 상태가 달라질 수 있음을 항상 염두에 둔다.
2. merge 충돌 해결 시 파일별 부모 커밋 의도를 확인하고 회귀가 없도록 처리한다.
3. 익숙하지 않은 변경이 보여도 먼저 의미를 파악하고, 되돌리기 전에 반드시 사용자 허가를 받는다.
4. 이름 변경이나 파일 이동이 포함된 변경은 diff만 보지 말고 실제 참조 관계와 히스토리 의미를 함께 본다.

## CRA 루프

사용자가 `CRA 루프`를 명시적으로 요청한 경우에만 아래 절차를 수행한다.

### 목적

CRA 루프에서 메인 에이전트의 역할은 리뷰 로그를 실시간으로 해석하는 것이 아니라, 리뷰 프로세스의 종료 상태를 확인하고 종료 후 결과를 검토하는 것이다.

### 상태 정의

리뷰 상태는 아래 네 가지로만 관리한다.

1. 실행 중
2. 종료: `no substantive findings` 또는 이에 준하는 최종 종료 문구 확인
3. 종료: substantive finding 존재
4. 실패: 에러 코드, 예외, 연결 실패, 프로세스 비정상 종료 등 명백한 실패 확인

### 실행 원칙

1. 리뷰는 반드시 blocking 방식으로 실행한다.
2. 백그라운드 실행(&)을 사용하지 않는다.
3. 프로세스 종료 자체를 리뷰 종료의 1차 기준으로 본다.
4. 필요 시 `review.done`과 같은 sentinel 파일을 활용하여 종료 여부를 판단할 수 있다.
5. 리뷰 명령은 종료 코드 확인이 가능하게 실행한다.
6. 중간 진행 상태를 보기 위해 동일 로그를 반복적으로 tail하지 않는다.
7. 리뷰 중에는 턴을 끝내지 않고 계속 붙어서 대기한다.
8. CRA 진행 중에는 final 응답을 내지 않고 commentary만 사용한다

예시:

```bash
codex review --commit <COMMIT_SHA> \
  -c model="gpt-5.4" \
  -c model_reasoning_effort="xhigh" \
  > review.log 2>&1
```

또는

```bash
rm -f review.done && codex review --commit <COMMIT_SHA> \
  -c model="gpt-5.4" \
  -c model_reasoning_effort="xhigh" \
  > review.log 2>&1 && touch review.done
```

### 상태 확인 원칙

1. 리뷰 실행 중에는 로그를 지속적으로 읽지 않는다.
2. 프로세스가 살아 있으면 추가 판단 없이 기다린다.
3. 상태 판단은 아래 순서를 따른다.

   * 프로세스 종료 여부
   * 종료 코드
   * sentinel 파일 존재 여부
   * 로그의 마지막 수십 줄
4. 로그는 1차 판단 수단이 아니다.

### 로그 소비 제한

1. 상태 확인 시에도 전체 로그를 읽지 않는다.
2. 마지막 50~100줄만 확인한다.
3. 동일한 로그를 반복해서 다시 읽지 않는다.
4. 중간 출력의 의미를 해석하지 않는다.
5. `substantive findings` 여부는 종료 후에만 판단한다.

### 절차

1. 작업 완료 후 변경사항을 commit 한다.
2. commit 직후 코드 리뷰를 요청한다.
3. 리뷰 실행 후, 프로세스 종료까지 별도의 해석 없이 기다린다.
4. 종료가 확인된 뒤 로그를 한 번만 확인한다.

### 리뷰 처리

리뷰가 종료된 이후에만 findings를 검토한다.

1. substantive finding이 존재하면

   * 타당성을 검토한다.
   * 유효하면 수정한다.
   * 수정 후 `git commit --amend --no-edit`로 갱신한다.
   * 다시 리뷰를 요청한다.

2. 반복

   * amend
   * review
   * 종료 확인

3. 종료 조건

   * `no substantive findings`
   * 또는 남은 지적이 모두 합리적으로 반박 가능한 상태

### 최종 보고

최종 보고에는 아래를 포함한다.

1. 최종 commit hash 및 작업 내용
2. CRA 중 반영한 리뷰 요약
3. 무효로 판단한 리뷰와 그 사유
4. 마지막 리뷰의 종료 상태
5. 실패 시 실패 원인
6. 이름 변경, 파일 이동, 문서 정합성 수정이 있었다면 그 이유와 영향 범위

### 핵심 원칙 정리

1. 리뷰는 스트리밍이 아니라 배치 작업이다.
2. 종료 전에는 어떤 판단도 하지 않는다.
3. 종료 후 한 번만 판단한다.

## 네트워크 및 권한

1. 네트워크 접근이 필요한 실행은 모두 사전에 식별한다.
2. sudo 명령은 사용자가 직접 실행한다.
3. 외부 API, 패키지 저장소, 원격 DB, 원격 캐시, 외부 SaaS 접근은 네트워크 접근으로 간주한다.
4. 권한이 부족한데도 우회 가능한 것처럼 가장하지 않는다.

## 문서

1. 코드 수정 후 docs/ 문서 반영 필요 여부를 항상 판단한다.
2. 문서와 구현이 어긋나면 함께 맞춘다.
3. 공개 동작이 바뀌었는데 문서가 그대로면 작업이 끝난 것이 아니다.
4. 함수명, 파일명, 모듈 구조, CLI 옵션, 설정 키가 바뀌면 관련 문서와 예제도 함께 갱신한다.
5. 이름 변경이 어렵다면 왜 유지하는지, 현재 의미가 무엇인지 문서에 남길지 검토한다.

## 이름 정합성 점검

1. 함수명은 현재 수행하는 핵심 동작을 반영해야 한다.
2. 파일명과 모듈명은 그 안의 주요 책임과 어긋나지 않아야 한다.
3. 초기 구현 당시에는 맞았지만 수정 이후 의미가 달라졌다면 rename, 분리, 이동 중 무엇이 맞는지 검토한다.
4. 아래 경우는 이름 정합성 점검을 반드시 수행한다.

   * 기존 함수의 책임이 확대되거나 축소된 경우
   * 파일 하나가 사실상 여러 책임을 갖게 된 경우
   * 함수가 더 이상 원래 자료형이나 원래 도메인만 처리하지 않는 경우
   * 예외 처리, 캐싱, 변환, 검증, 외부 호출이 추가되어 의미가 바뀐 경우
5. 이름 변경 시에는 호출부, 테스트명, 문서, 로그 메시지, 메트릭명, 설정명까지 함께 점검한다.
6. 외부 계약 때문에 이름을 유지해야 하면 내부 주석, 문서, 래퍼 함수, deprecated 안내 등으로 의미 불일치를 완화할 방법을 검토한다.

## 미래의 작업자를 위한 개선

1. 프로젝트 경로의 AGENTS.md는 미래의 작업자에게도 유용해야 한다.
2. 반복적으로 발생하는 실수, 배포 함정, 운영 제약이 확인되면 AGENTS.md 업데이트를 제안하거나 반영한다.
3. 특히 CRA 루프에서 반복적으로 발생하는 낭비(중간 로그 과소비, 종료 상태 오판, 불필요한 재해석)가 확인되면 이를 방지하는 규칙을 보강한다.
4. 반복된 실수나 혼동이 두 번 이상 재발하면 로컬 규칙, 체크리스트, 명명 규칙, 검증 절차 중 어디에 반영할지 검토한다.
5. “코드는 맞지만 이름과 문서가 틀린 상태”가 반복되면 이를 독립적인 실패 유형으로 간주하고 규칙을 보강한다.
