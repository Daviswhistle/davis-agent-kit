# AGENTS.md for Korean

한국어로 작업하는 터미널 기반 AI 코딩 에이전트를 위한 전역 지시문 모음입니다.

이 저장소는 원래 Codex CLI용 `AGENTS.md`를 관리하기 위해 만들었지만, 현재는 Gemini CLI용 `GEMINI.md`도 함께 제공합니다. 두 파일은 같은 철학을 공유하지만, 각 CLI의 지시 파일 탐색 방식, 메모리 로딩 방식, 리뷰 실행 방식이 다르기 때문에 완전히 같은 파일을 이름만 바꿔 쓰는 구조는 아닙니다.

## 목적

이 저장소의 목적은 에이전트가 매번 아래 기준을 일관되게 따르도록 만드는 것입니다.

- 사용자와의 대화는 한국어로 진행한다.
- 모호하거나 비현실적인 요청은 그대로 따르지 않고 먼저 문제를 지적한다.
- 임시방편보다 근본 원인 해결을 우선한다.
- 불확실한 외부 동작, 라이브러리 동작, 운영 환경 특성은 추정하지 않고 확인한다.
- 코드, 문서, 테스트, 설정, 배포 방식의 정합성을 함께 본다.
- 이름을 유지보수 인터페이스의 일부로 보고, 함수명/파일명/문서명/테스트명이 실제 역할과 어긋나면 수정 후보로 본다.
- 불필요하게 전체 로그나 거대한 파일을 소비하지 않고, 필요한 범위부터 확인한다.
- CRA/TCA처럼 커밋 단위 검증이 필요한 작업은 작은 작업 단위로 나누고, 리뷰가 끝나기 전 다음 작업으로 넘어가지 않는다.

## 파일 구성

| 파일 | 대상 | 용도 |
| --- | --- | --- |
| `AGENTS.md` | Codex CLI | Codex가 읽는 전역/프로젝트 지시문 |
| `GEMINI.md` | Gemini CLI | Gemini CLI가 읽는 전역/프로젝트 지시문 |

## 어떤 파일을 써야 하나

Codex CLI를 쓴다면 `AGENTS.md`를 사용합니다.

Gemini CLI를 쓴다면 `GEMINI.md`를 사용합니다.

둘 다 쓴다면 두 파일을 모두 설치하는 것이 가장 안전합니다. 두 CLI가 지시문을 읽는 방식이 다르고, 특히 리뷰 루프에서 사용하는 명령과 출력 형식이 다르기 때문입니다.

## 설치 방법

### Codex CLI 전역 지시문으로 사용

```bash
mkdir -p ~/.codex
cp AGENTS.md ~/.codex/AGENTS.md
```

확인:

```bash
codex --ask-for-approval never "Summarize the current instructions."
```

Codex는 기본적으로 `~/.codex/AGENTS.md`를 전역 지시문으로 읽습니다. `CODEX_HOME`을 따로 지정했다면 해당 경로 아래의 `AGENTS.md`가 기준이 됩니다.

### Gemini CLI 전역 지시문으로 사용

```bash
mkdir -p ~/.gemini
cp GEMINI.md ~/.gemini/GEMINI.md
```

Gemini CLI 안에서 확인:

```text
/memory show
```

지시 파일을 수정한 뒤 다시 로드하려면 다음을 사용합니다.

```text
/memory refresh
```

일부 문서나 버전에서는 `/memory reload`라는 표현도 보일 수 있습니다. 현재 공식 명령 문서 기준으로는 `/memory refresh`가 계층형 메모리를 다시 로드하는 명령입니다.

## 프로젝트별 지시문으로 사용

전역 지시문이 아니라 특정 저장소에만 적용하고 싶다면 프로젝트 루트에 파일을 둡니다.

Codex CLI:

```bash
cp AGENTS.md /path/to/project/AGENTS.md
```

Gemini CLI:

```bash
cp GEMINI.md /path/to/project/GEMINI.md
```

프로젝트별 파일은 해당 프로젝트의 언어, 테스트 명령, 배포 방식, 금지 작업, 디렉터리별 예외를 추가하는 데 적합합니다.

## Codex와 Gemini의 차이

| 항목 | Codex CLI | Gemini CLI |
| --- | --- | --- |
| 기본 지시 파일 | `AGENTS.md` | `GEMINI.md` |
| 전역 위치 | `~/.codex/AGENTS.md` | `~/.gemini/GEMINI.md` |
| 프로젝트 위치 | 프로젝트 루트와 하위 디렉터리의 `AGENTS.md` 또는 `AGENTS.override.md` | 프로젝트 루트, 상위/하위 디렉터리의 `GEMINI.md` |
| 활성 지시 확인 | Codex에게 현재 instruction source 요약을 요청하거나 로그 확인 | `/memory show`, `/memory list` |
| 지시문 재로드 | 새 세션 또는 새 실행에서 다시 탐색 | `/memory refresh` |
| 리뷰 루프 | `codex review`를 사용할 수 있음 | `gemini -p` 기반의 읽기 전용 리뷰 프롬프트로 구성 |

## Gemini를 추가할 때 단순히 리뷰 예시만 바꾸면 되는가

아닙니다.

핵심 철학은 공유할 수 있지만, Gemini용 파일에서는 아래 차이를 반영해야 합니다.

1. `GEMINI.md`는 Gemini CLI의 계층형 메모리로 로드됩니다.
2. 현재 로드된 지시는 `/memory show`와 `/memory list`로 확인하는 편이 안전합니다.
3. 파일을 수정한 뒤에는 `/memory refresh`로 다시 로드해야 합니다.
4. Codex의 `codex review`와 같은 전용 리뷰 명령을 그대로 전제하지 말고, `gemini -p`에 읽기 전용 리뷰 프롬프트를 명시하는 방식으로 운영합니다.
5. 리뷰어 역할의 Gemini가 파일을 수정하지 않도록 프롬프트에 “읽기 전용” 제약을 반복해서 명시해야 합니다.
6. `--model`이나 `/model`이 모든 하위 실행이나 sub-agent까지 강제한다고 가정하지 않는 편이 안전합니다.
7. Gemini는 출력 형식을 프롬프트로 강하게 고정해야 하므로, `REVIEW_STATUS: NO_SUBSTANTIVE_FINDINGS` 같은 종료 토큰을 명확히 두는 것이 좋습니다.

## CRA 루프 개요

CRA는 Commit Review Agent 루프입니다.

비사소한 변경을 하나의 커밋으로 만든 뒤, 별도 리뷰어 에이전트에게 읽기 전용 리뷰를 맡기고, substantive finding이 있으면 수정 후 amend하고 다시 리뷰합니다.

핵심 원칙은 다음과 같습니다.

- 리뷰는 스트리밍 해석 대상이 아니라 배치 작업이다.
- 중간 로그를 반복해서 읽지 않는다.
- 종료 코드, sentinel 파일, 마지막 로그 일부만 확인한다.
- substantive finding은 종료 후에만 판단한다.
- 유효한 지적만 반영한다.
- 반영 후에는 국소 검증부터 다시 수행한다.

## TCA 루프 개요

TCA는 Task-Commit-Approve 루프입니다.

큰 작업을 독립적으로 검증 가능한 작은 작업 단위로 나누고, 각 작업 단위마다 아래 순서를 지킵니다.

1. Task: 하나의 명확한 작업 단위를 수행한다.
2. Commit: 해당 작업 단위만 커밋한다.
3. Approve: CRA로 커밋 단위 리뷰를 수행한다.
4. 통과 후 다음 작업으로 넘어간다.

이 방식은 여러 목적의 변경이 하나의 커밋에 섞이는 문제, 리뷰 반영 중 작업 경계가 흐려지는 문제, 테스트 실패 원인이 불명확해지는 문제를 줄이기 위한 운영 규칙입니다.

## 권장 운영 방식

1. 전역 파일에는 모든 프로젝트에 적용할 원칙만 둡니다.
2. 프로젝트별 테스트 명령, 빌드 명령, 배포 규칙은 프로젝트 루트의 `AGENTS.md` 또는 `GEMINI.md`에 둡니다.
3. 디렉터리별 예외가 크면 하위 디렉터리에 별도 지시 파일을 둡니다.
4. 지시문이 길어지면 중요한 원칙이 잘리지 않는지 확인합니다.
5. 실제 적용 여부는 반드시 CLI 안에서 확인합니다.
6. 모델 이름, sandbox, 승인 모드, 네트워크 접근, shell 권한은 세션마다 달라질 수 있으므로 작업 시작 시 먼저 확인합니다.

## 주의사항

이 저장소의 파일은 강한 작업 규칙을 담고 있습니다. 따라서 아래 상황에서는 그대로 복사하기보다 프로젝트에 맞게 줄이는 편이 좋습니다.

- 짧은 개인 스크립트만 다루는 프로젝트
- 테스트/빌드 체계가 거의 없는 저장소
- 커밋 단위 리뷰 루프가 과한 소규모 작업
- 영어 커뮤니케이션이 필요한 팀 저장소
- 에이전트에게 많은 재량을 주고 빠른 실험을 우선하려는 경우

반대로 아래 상황에서는 이 파일이 특히 유용합니다.

- 장기 유지보수 프로젝트
- 자동화 에이전트가 반복적으로 작업하는 저장소
- 테스트, 문서, 배포 설정의 정합성이 중요한 코드베이스
- 리뷰 로그나 대량 로그를 무분별하게 읽는 문제를 줄이고 싶은 환경
- 커밋 단위로 작업을 검증하고 싶은 환경
