# 전역 AGENTS.md

## 기본 원칙
1. 사용자와의 대화는 한국어로 한다.
2. 사용자 지시가 모호하거나, 현실을 충분히 반영하지 못하거나, 기술적으로 비합리적이라고 판단되면 그 점을 분명히 말한다.
3. 항상 최선을 다한다. 임시방편보다 근본 원인 해결을 우선한다.
4. 문제를 우회하여 겉으로만 해결된 것처럼 보이게 하지 않는다.
5. 작업 범위를 불필요하게 넓히지 않는다. 다만, 수정 범위와 직접 연결된 회귀, 불일치, 불필요 코드는 함께 정리한다.
6. 다른 작업자의 변경으로 보이는 내용은 임의로 되돌리지 않는다.
7. 기본적으로 commit과 push는 하지 않는다.
8. 단, 사용자가 `CRA 루프`를 명시적으로 요청한 경우에만 commit/amend를 허용한다.
9. sudo 권한이 없다는 전제를 유지한다. 필요한 경우 사용자에게 직접 실행을 요청한다.

## 작업 시작 전 절차
1. 먼저 현재 작업의 목표를 한 문장으로 정리한다.
2. 관련 파일, 모듈, 설정, 배포 파일, 문서 위치를 확인한다.
3. 변경으로 인해 영향을 받을 수 있는 범위(회귀 가능성, 운영 영향, 배포 영향, 데이터 영향)를 먼저 점검한다.
4. DB 작업이 포함되면 .sql 마이그레이션 파일을 먼저 확인하여 설계 의도와 맞는지 검토한다.
5. 네트워크 접근 또는 권한 상승이 필요한 실행은 사전에 식별한다.
6. 외부 지식이 불확실하면 웹 검색으로 확인한 뒤 작업한다.

## 구현 원칙
1. 처음 보는 사람도 이해할 수 있도록 주석을 친절하고 자세하게 한국어로 작성한다.
2. 주석에는 AI가 작업했다는 흔적이나 사용자와의 대화 맥락을 남기지 않는다.
3. 코드를 수정할 때에는 기존 주석과 실제 동작이 계속 일치하는지 함께 점검한다.
4. 유지보수와 확장에 유리하면 함수, 모듈 단위로 구조화한다.
5. 모듈 위치는 프로젝트 전체 구조와 일관되게 배치한다.
6. 변수명, 함수명, 타입명은 처음 보는 사람도 역할을 추론할 수 있게 짓는다.
7. 비슷한 역할의 함수가 중복되면 통합 가능성을 검토하고, 바로 합치지 않더라도 사용자에게 제안한다.
8. 불필요한 코드와 사용처는 함께 정리한다.
9. Python 파일을 수정할 때는 기존 들여쓰기 형식을 먼저 확인하고 동일한 형식을 따른다.

## 리뷰 처리 원칙
1. 리뷰가 들어오면 맹신하지 않는다.
2. 각 리뷰 항목에 대해 먼저 아래를 검토한다.
   - 사실관계가 맞는가
   - 현재 작업 범위에 해당하는가
   - 실제 운영 리스크가 있는가
   - 반영 시 더 큰 회귀를 만들지 않는가
3. 타당한 리뷰만 반영한다.
4. 타당하지 않은 경우 리뷰어가 오해하지 않도록 주석을 충분히 작성한다.
5. 리뷰 로그의 중간 출력은 참고 정보일 뿐이며, 최종 판정은 명시적 종료 상태와 최종 결과를 기준으로 한다.

## 작업-리뷰-검증-반영 루프
비사소한 수정은 아래 순서를 따른다.

1. 계획 수립
   - 작업 목표
   - 영향 파일
   - 유지되어야 할 동작
   - 예상 리스크
2. 구현
   - 가장 작은 단위의 일관된 수정부터 적용
3. 자체 리뷰 또는 리뷰 반영 검토
   - 정확성
   - 회귀 가능성
   - 예외, 경계값
   - 성능
   - 가독성, 유지보수성
   - 기존 주석 및 문서와의 일치 여부
4. 검증
   - 가능한 테스트, 린트, 타입체크, 빌드를 수행한다.
   - 실행하지 못한 검증이 있으면 그 이유를 명시한다.
5. 반영
   - 검증된 리뷰 지적만 반영한다.
6. 재검증
   - 수정 후 다시 테스트, 린트, 타입체크, 빌드, 리뷰를 수행한다.
7. 종료
   - 더 이상 유효한 중대 지적이 없고
   - 현재 작업 범위에서 필요한 검증이 완료되었을 때 종료한다.

## 종료 조건
아래 조건을 모두 만족할 때 작업을 마친다.
1. 남아 있는 valid critical/high 이슈가 없다.
2. 최신 검토에서 새로운 valid medium 이슈가 나오지 않는다.
3. 실행 가능한 테스트, 린트, 타입체크, 빌드가 끝났다. 실행하지 못한 것은 이유를 명시했다.
4. 변경 사항, 남은 리스크, 반영하지 않은 리뷰 항목을 설명할 수 있다.
5. 사용자가 `CRA 루프`를 요청한 경우, 마지막 리뷰의 명시적 종료 상태를 확인했다.

## 금지 사항
1. git push를 하지 않는다.
2. 사용자의 허가 없이 git restore, git checkout, reset, revert, 수동 덮어쓰기 등 이전 상태로 되돌리는 작업을 하지 않는다.
3. 다른 작업자의 변경을 임의로 제거하지 않는다.
4. 문제의 본질을 바꾸어 겉으로만 해결된 것처럼 만들지 않는다.
5. `CRA 루프` 중 리뷰 프로세스가 명시적으로 종료되지 않았는데 임의 판단으로 완료 처리하지 않는다.
6. 리뷰가 실행 중이라는 이유만으로 중간 출력을 반복적으로 읽거나 해석하여 토큰과 문맥을 과도하게 소비하지 않는다.
7. 동일한 리뷰 중간 로그를 여러 번 다시 읽으며 의미를 재판단하지 않는다.
8. 환경변수를 사용하는 작업을 수행할 경우 터미널에 그 값이 출력되지 않도록 한다.

## Git 및 충돌 처리
1. 작업 중 저장소 상태가 달라질 수 있음을 항상 염두에 둔다.
2. merge 충돌 해결 시 파일별 부모 커밋 의도를 확인하고 회귀가 없도록 처리한다.
3. 익숙하지 않은 변경이 보여도 먼저 의미를 파악하고, 되돌리기 전에 반드시 사용자 허가를 받는다.

## CRA 루프
사용자가 `CRA 루프`를 명시적으로 요청한 경우에만 아래 절차를 수행한다.

### 목적
CRA 루프에서 메인 에이전트의 역할은 리뷰 로그를 실시간으로 해석하는 것이 아니라, 리뷰 프로세스의 종료 상태를 확인하고 종료 후 결과를 검토하는 것이다.

### 상태 정의
리뷰 상태는 아래 네 가지로만 관리한다.
1. 실행 중
2. 종료: `no substantive findings` 또는 이에 준하는 최종 종료 문구 확인
3. 종료: substantive finding 존재
4. 실패: 에러 코드, 예외, 연결 실패, 프로세스 비정상 종료 등 명백한 실패 확인

실행 중 상태에서는 리뷰 내용의 의미를 계속 해석하지 않는다. 종료 후에만 findings의 타당성을 검토한다.

### 절차
1. 작업 완료 후 변경사항을 commit 한다.
2. commit 직후 코드 리뷰를 요청한다.
   - `codex review --commit <COMMIT_SHA> -c model="gpt-5.4" -c model_reasoning_effort="xhigh"` 명령으로 리뷰를 요청한다.
   - 가능하면 리뷰 출력은 파일로 리다이렉트하거나 별도 로그 파일로 저장한다.
   - 메인 에이전트는 리뷰의 스트리밍 출력을 계속 따라가며 읽지 않는다.
   - 메인 에이전트는 리뷰 로그 전체를 지속적으로 해석하지 않고, 종료 상태 판정에 필요한 최소 정보만 확인한다.
   - 에러 등 비정상 요인이 아니라면 리뷰가 오래 걸리더라도 끈기 있게 기다린다.
   - 리뷰 명령이 실행 중이면 임의로 중단하거나, “충분히 검토된 것 같다”는 판단만으로 CRA 루프를 종료하지 않는다.
   - 다음 둘 중 하나가 명시적으로 확인되기 전에는 리뷰가 끝난 것으로 간주하지 않는다.
     - `substantive findings` 또는 `no substantive findings`와 동등한 최종 종료 문구가 출력되었다.
     - 리뷰 프로세스의 명백한 실패가 확인되었다. 예: 에러 코드, 예외, 연결 실패, 프로세스 비정상 종료
        - `write_stdin failed: stdin is closed for this session` 등 리뷰 중 리뷰어의 명령어가 샌드박스로 인해 차단된 상황 등은 정상적인 리뷰 과정이므로 리뷰 프로세스의 실패로 보지 않는다.
   - 출력이 한동안 없더라도 프로세스가 살아 있으면 계속 기다린다.
   - 장시간 무출력 상태에서는 아래만 확인한다.
     - 프로세스가 아직 실행 중인지
     - 로그의 마지막 수십 줄에 최종 종료 문구 또는 실패 신호가 있는지
   - 상태 확인 시에는 전체 로그가 아니라 마지막 50~100줄만 본다.
   - 새 출력이 없으면 이전 로그를 다시 분석하지 않는다.
   - 동일한 중간 출력 로그를 반복해서 다시 읽지 않는다.
   - substantive finding 판정은 리뷰가 명시적으로 종료된 뒤, 최종 로그 기준으로만 수행한다.
3. 리뷰에서 substantive finding이 나오면 타당성을 먼저 검토한다.
   - 유효한 지적이면 수정한다.
   - 수정이 있으면 기존 commit을 `git commit --amend --no-edit`로 갱신한다. 필요하면 커밋 메시지도 수정한다.
   - 같은 방식으로 다시 리뷰를 요청한다.
   - 무효라고 판단한 지적은 왜 무효인지 최종 보고에 간단히 남긴다.
4. `no substantive findings`가 나오거나, 리뷰가 명시적으로 종료된 뒤 남은 지적이 모두 무효라고 합리적으로 설명된 상태가 될 때까지 `amend -> re-review`를 반복한다.
   - CRA 진행 중에는 턴을 종료하지 않는다.
5. 최종 보고에는 아래를 포함한다.
   - 최종 commit hash 및 최종 commit 기준 작업 사항
   - 반영한 리뷰 지적 요약
   - 무효라고 판단한 지적이 있으면 그 사유
   - 마지막 리뷰가 `no substantive findings`인지 여부
   - 마지막 리뷰가 비정상 종료였다면, 어떤 형태의 실패였는지

### 리뷰 상태 확인 예시
리뷰 실행 중에는 아래와 같은 형태로 상태를 확인한다.
- 프로세스 생존 여부 확인
- 로그 파일의 마지막 일부 확인
- 최종 종료 문구 또는 실패 신호가 있는지 확인

리뷰 중간 출력 전체를 지속적으로 소비하거나, 중간 reasoning을 실시간으로 따라가며 해석하지 않는다.

## 네트워크 및 권한
1. 네트워크 접근이 필요한 실행은 모두 사전에 식별하고, 필요 시 권한 상승이 필요하다는 점을 사용자에게 알린다.
2. sudo 명령은 사용자가 직접 실행해야 한다.

## 문서
1. 코드 수정 후 docs/ 문서 반영 필요 여부를 항상 판단한다.
2. 문서와 구현이 어긋나면 함께 맞춘다.

## 미래의 작업자를 위한 개선
1. 프로젝트 경로의 AGENTS.md는 미래의 작업자에게도 유용해야 한다.
2. 반복적으로 발생하는 실수, 배포 함정, 운영 제약이 확인되면 AGENTS.md 업데이트를 제안하거나 반영한다.
3. 특히 CRA 루프에서 반복적으로 발생하는 낭비(예: 중간 로그 과소비, 불필요한 재해석, 종료 상태 오판)가 확인되면, 이를 방지하는 규칙을 문서에 보강한다.
